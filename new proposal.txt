🎯 1. PURPOSE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Introduce an AI-Powered Proposal Module that allows Administrators to automatically
generate draft rosters based on recurring tasks, member skills, availability, and
fairness constraints.

The AI agent uses Microsoft AI Agent Framework + GitHub Models (free tier), so there
is ZERO cost during development. The model name and GitHub PAT token are read from
appsettings — never hardcoded. Generation runs ASYNCHRONOUSLY so the UI is never
blocked.

This module integrates with the existing Clean Architecture and Multi-Tenancy design.
Published proposals trigger the standard assignment workflow (Notifications,
Accept/Reject). This module is ADMIN ONLY — members cannot access proposal pages.

✅ Cost: GitHub Models free tier — zero cost for development and light production use.
✅ No rule-based fallback strategy needed — agent-only approach via GitHub Models.

🤖 2. AI AGENT DESIGN (Confirmed)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Framework:     Microsoft.Extensions.AI (Microsoft AI Agent Framework)
Model API:     GitHub Models — https://models.inference.ai.azure.com
Auth:          GitHub Personal Access Token (PAT)
Config source: appsettings.Development.json → "GitHubModels" section
               (empty placeholders live in appsettings.json for production override)

Config structure (appsettings.Development.json):
  "GitHubModels": {
    "Endpoint": "https://models.inference.ai.azure.com",
    "ModelName": "gpt-4o",
    "Token":     "ghp_YOUR_PAT_HERE"
  }

Agent pattern:
  - Agent receives scheduling rules and constraints as a SYSTEM PROMPT
    (e.g. "distribute tasks fairly, respect monthly limits, check skill requirements")
  - Agent calls C# TOOL FUNCTIONS to fetch real data and write proposal items
  - No rule-based fallback — GitHub Models free tier handles all generation

Agent tool functions (C# async methods registered with the agent):
  1. GetRecurringTasksAsync(tenantId)                          → task list + day rules
  2. GetQualifiedMembersAsync(tenantId, taskId)                → members with required skill
  3. GetMemberAssignmentCountAsync(userId, month, year)        → fairness data
  4. GetExistingAssignmentsAsync(tenantId, startDate, endDate) → conflict awareness
  5. CreateProposalItemAsync(proposalId, taskId, userId, date) → writes draft item to DB
  6. LogSkippedSlotAsync(proposalId, taskId, date, reason)     → audit trail for gaps

📋 5. FEATURE SPECIFICATIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

5.1  Generate Proposal (Async)
  1. Admin POSTs /api/v1/proposals with { name, dateRangeStart, dateRangeEnd }
  2. API creates RosterProposal with Status = Processing, returns HTTP 202 + { proposalId } immediately
  3. ProposalId is queued on Channel<Guid> — no blocking of the HTTP thread
  4. Background IHostedService dequeues and runs ProposalAgentService.GenerateAsync(...)
       - System prompt encodes scheduling rules (fairness, skill checks, monthly limits)
       - Agent loop calls tool functions to fetch data and write items
  5. On completion: Status → Draft
  6. Frontend polls GET /api/v1/proposals/{id} every 3 s, watches Status field
     (shows "Processing…" spinner until status becomes "Draft")
  7. Only ONE proposal per tenant may be in Processing status at a time
     (enforced in GenerateProposalCommand handler — returns 409 if another is in-flight)

5.2  Preview & Edit Proposal
  - Admin views all items grouped by date
  - Admin can:
      Edit item   → change assigned member (PATCH item)
      Add item    → pick task + member + date (POST item)
      Delete item → remove a slot (DELETE item)
  - All edits require Status = Draft (reject with 409 if Processing or Published)

5.3  Publish Proposal
  - Admin clicks Publish (requires Status = Draft)
  - For each RosterProposalItem:
      If a live Assignment already exists for same TenantId + TaskId + EventDate:
          → mark item Skipped, write ProposalSkipLog, CONTINUE (never fail the batch)
      Else:
          → create Assignment (Status = Pending), trigger push/email notification
  - Set RosterProposal.Status = Published
  - Published proposals are permanent history (never deleted, visible in /proposals)

5.4  Print Draft PDF
  - Reuses QuestPDF pattern from existing ReportService
  - Same layout as Monthly Roster PDF
  - Large diagonal "DRAFT" watermark across every page
  - Items grouped by date
  - Available for both Draft and Published proposals

5.5  Dashboard Widget
  - Shows: count of Draft + Processing proposals
  - Shows: most recent proposal name + generated date
  - Quick-link to /proposals page

🗄️ 6. DATABASE SCHEMA
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

CREATE TABLE roster_proposals (
    proposal_id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id           UUID NOT NULL REFERENCES tenants(tenant_id),
    name                VARCHAR(255) NOT NULL,
    status              VARCHAR(50)  NOT NULL DEFAULT 'Processing',
    date_range_start    DATE NOT NULL,
    date_range_end      DATE NOT NULL,
    generated_at        TIMESTAMP DEFAULT NOW(),
    published_at        TIMESTAMP,
    created_by_user_id  INTEGER NOT NULL
);

CREATE TABLE roster_proposal_items (
    item_id      UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    proposal_id  UUID NOT NULL REFERENCES roster_proposals(proposal_id) ON DELETE CASCADE,
    task_id      INTEGER NOT NULL REFERENCES tasks(task_id),
    user_id      INTEGER NOT NULL REFERENCES users(user_id),
    event_date   DATE NOT NULL,
    status       VARCHAR(50) NOT NULL DEFAULT 'Proposed',
    skip_reason  TEXT
);

CREATE TABLE proposal_skip_logs (
    log_id       UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    proposal_id  UUID NOT NULL REFERENCES roster_proposals(proposal_id) ON DELETE CASCADE,
    task_id      INTEGER NOT NULL,
    event_date   DATE NOT NULL,
    reason       TEXT NOT NULL,
    logged_at    TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_roster_proposals_tenant  ON roster_proposals(tenant_id);
CREATE INDEX idx_proposal_items_proposal  ON roster_proposal_items(proposal_id);
CREATE INDEX idx_skip_logs_proposal       ON proposal_skip_logs(proposal_id);
